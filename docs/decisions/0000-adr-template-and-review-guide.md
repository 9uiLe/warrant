## なぜ「型」なのか {#why}

Markdown の散文は **スラスラ読めてしまう**。この「読みやすさ」を脳は「理解した」と取り違える（*処理流暢性 / 説明深度の錯覚*）。結果、論理の飛躍・前提の曖昧さ・選択肢比較の甘さを見落としたまま承認し、後から『なんか違う』が起きる。**スラスラ読めること自体がレビューの罠** だった。

これを塞ぐのは保存形式ではなく **型** — 固定スロットと、論拠を強制する欄。そしてそれを **摩擦のある描画**（生成 HTML を viewer で読む）と組み合わせる。形式そのものの議論は [ADR-0002](0002-adr-source-format.html) で決着している。

> [!INFO] 流し読みを防ぐ3つの作用
> - **構造の明示** — 埋まっていないセクションや薄い論拠が「見えて」しまう。
> - **読む摩擦** — diff をスキャンで流さず、文書として向き合う。
> - **知識への紐付け** — リンク・アンカー・表で技術/ドメイン知識に接続できる。

## 固定スロット・テンプレート {#template}

すべての ADR は次のスロットを持つ。各スロットは「どの流し読み失敗を塞ぐためにあるか」と対になっている。空にできるが、**空であること自体が論点として見える** のが要点。

| スロット | 書くこと | 塞ぐ失敗 |
|---|---|---|
| Context | なぜ今この決定が要るか。前提・制約・環境（できれば実測値）。 | 「何を決めているか」が曖昧なまま進む。 |
| Decision Drivers | 判断基準を箇条書き。後段の比較・結論はこの語彙で書く。 | 結論の後付け正当化。基準が動いて見える。 |
| Considered Options | 最低 2 案。各案の長所 **と短所**。採用案にも短所を書く。 | 本命だけ厚く書き、対案を藁人形にする。 |
| 比較表 | 判断軸 × 各案。採用列のヘッダに `[*]`。セルは `yes`/`no`/`mid`/`na`。 | 散文で「なんとなく良い」と流す。軸ごとの優劣が曖昧。 |
| Decision | `> [!SUCCESS]` で採用案と理由。Drivers の語彙で根拠を述べる。 | 「結局どうするのか」がぼやける。 |
| Consequences | Positive / **Negative** / Neutral。採用案の代償を具体的に。 | 良い面だけ並べ、代償を直視しない。 |
| 反証（間違いになるとしたら） | **観測可能なトリガー** つきで、前提が崩れる条件を列挙。 | 「状況が変われば見直す」式の空文句で逃げる。 |
| Compliance & Monitoring | 決定の遵守をどう担保・観測するか（任意）。 | 決めただけで運用に落ちない。 |
| References | 根拠リンク・関連 ADR・規格・コード。 | 主張が知識に接続されず宙に浮く。 |

## 反証スロット — 「この決定が間違いになるとしたら、何が原因か」 {#falsification}

これがこの型の **核心**。承認者は「正しいか？」を確かめにいくと流暢性に負ける。逆に **「どう間違いうるか」** を書かせると、暗黙の前提が露出する。

> [!TIP] 良い反証スロットの条件
> - **暗黙の前提を名指しする**（例: 「人間がローカルのブラウザで読む」前提に立っている）。
> - 各前提に **観測可能なトリガー** を添える（「〜という要望が複数回出たら」）。漠然とした「状況次第」は不可。
> - 崩れたときの **代替の方向** を一言で示す（「その時は AsciiDoc を検討」）。

実例: [ADR-0001 の反証スロット](0001-local-adr-viewer.html#falsification)・[ADR-0002 の反証スロット](0002-adr-source-format.html#falsification) — いずれも観測指標つきで「崩れる条件」を並べている。

## レビュー観点チェックリスト {#checklist}

承認前に、レビュアーは各項目を **反証しにいく**。どれか 1 つでも「言えない／一致しない」なら、まだ承認しない。これが流し読みを破る最後の関門。

### 構造 {#check-structure}

- 固定スロットが **すべて存在** するか。空のスロットは「不要だから空」なのか「埋め忘れ」なのか本文で判別できるか。
- この ADR が **決めている一文** を、Decision を見ずに Context だけから言えるか。

### 論理 {#check-logic}

- 各 Option について「**却下する具体的理由**」を 1 行で言えるか。言えないなら比較が甘い（藁人形になっていないか）。
- Decision の根拠は **Decision Drivers の語彙** で書かれているか。Drivers に無い基準が結論で初登場していないか（後付け）。
- 比較表の各セルは **本文の主張と一致** するか。表だけ採用案を良く見せていないか。`mid`/`no` の根拠が本文にあるか。
- 採用案の **Negative** が具体的に書かれているか。「特になし」は赤信号。

### 前提と反証 {#check-assumption}

- **暗黙の前提** が最低 1 つ明示されているか。反証スロットに **観測可能なトリガー** があるか（「状況が変われば」式の空文句でないか）。
- その前提が **今この瞬間に成り立っている** 確証はあるか（思い込みでないか）。

### 知識への接続 {#check-knowledge}

- 主要な主張に **References / 関連 ADR / 規格 / コード** への接続があるか。宙に浮いた断定がないか。
- 影響範囲（他の ADR・既存実装）への **波及** を見落としていないか。

### 流暢性の罠 {#check-fluency}

- 「読みやすくて納得した」だけで通そうとしていないか。**納得の根拠を 1 つ、自分の言葉で再構成** できるか。できないなら、それは理解でなく流暢性。

## 運用 {#ops}

- **執筆**: 正本は `adr/NNNN-slug.md`（制約付き Markdown、規約は `tools/adr-format.md`）。
- **生成**: `node tools/adr-gen.mjs` で HTML 生成。`node tools/adr-lint.mjs` でスロット充足を検査。
- **閲覧・承認**: `node tools/adr-view.mjs` でローカル配信（一覧・ライブリロード付き）。承認は生 diff でなく **viewer の描画** に対して行う（[ADR-0002](0002-adr-source-format.html)）。
- **採番**: `NNNN-kebab-title.md`。`0000` は本ガイド（テンプレート/規約）に予約。
- **Status**: `proposed` / `accepted` / `deprecated` / `superseded` / `rejected`。

## このチェックリストを鋭くするために {#input-needed}

> [!WARNING] 未反映の入力：過去の「なんか違う」実例
> 現状のチェックリストは一般原則から組んでいる。**実際に過去に『なんか違う』が起きた具体箇所**（前提漏れ／選択肢比較の甘さ／影響範囲の見落とし 等）を 2〜3 件もらえれば、その失敗を名指しで潰す項目を追加し、このリストを **あなたのチームの失敗パターンに特化** させられる。
